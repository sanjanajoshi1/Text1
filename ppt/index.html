What is Version Control?

A Version Control System (VCS) is a tool that helps a software team manage changes to source code over time.


  • Track Changes: See who made what changes and when.
  • Revert to Previous Versions: Easily undo mistakes or go back to a stable state.
  • Collaborate: Allows multiple developers to work on the same project simultaneously without overwriting each other's work.
  • Branching and Merging: Work on new features in isolation (branches) and then combine them back into the main project (merge).

What is Git?

Git is a Distributed Version Control System (DVCS). This means that every developer has a full copy of the project history on their local machine.

Fast

Most operations are performed locally, making it incredibly fast.

Branching

Git's branching model is lightweight and powerful, encouraging feature-based workflows.

Distributed

No single point of failure. You can work offline and sync changes later.

Secure

All changes are cryptographically hashed (SHA-1), ensuring data integrity.

The Three States of Git

Your files can be in one of three main states:

1. Working Directory

The single checkout of one version of the project. These files are pulled out of the compressed database in the Git directory and placed on disk for you to use or modify.

2. Staging Area (Index)

A file in your Git directory that stores information about what will go into your next commit. You add snapshots of your modified files to the staging area.

3. Git Directory (.git)

Where Git stores the metadata and object database for your project. This is the most important part of Git, and it is what is copied when you clone a repository.

Core Git Commands

# Initialize a new Git repository in the current directory $ git init # Add a file to the staging area $ git add <file_name> # Add all modified files to the staging area $ git add . # Commit the staged changes with a message $ git commit -m "Your descriptive commit message" # Check the status of your working directory and staging area $ git status

Working with Remotes

# Clone (download) a repository that exists on GitHub $ git clone <repository_url> # Add a remote repository (e.g., from GitHub) $ git remote add origin <repository_url> # Push your committed changes to a remote repository $ git push origin <branch_name> # Pull changes from a remote repository $ git pull origin <branch_name> # See the history of commits $ git log

What is GitHub?

GitHub is a web-based platform that provides hosting for software development and version control using Git. It's like a "cloud for your code."


  • Remote Repositories: A central place to store and manage your Git repositories.
  • Collaboration Tools: Features designed to make teamwork seamless.
  • Project Management: Tools to track issues, plan features, and manage your project's lifecycle.
  • Automation: GitHub Actions allows you to automate your build, test, and deployment pipeline.

GitHub's Key Features

Pull Requests

Propose changes to a repository. They let you tell others about changes you've pushed to a branch and request their feedback before merging into the main branch.

Code Review

Discuss proposed changes directly in the pull request. You can comment on specific lines of code, suggest improvements, and approve changes.

Issues

Track bugs, enhancements, and other tasks. Issues are a great way to keep track of tasks, enhancements, and bugs for your projects.

Actions

Automate your workflow. Build, test, and deploy your code right from GitHub. Make code reviews, branch management, and issue triaging work the way you want.

A Typical GitHub Workflow

  1. 1

    Create a Branch

    Create a new branch from `main` to work on a new feature or bug fix.

    git checkout -b new-feature
  2. 2

    Add Commits

    Make your changes, add them to staging, and commit them with clear messages.

    git add . && git commit -m "Implement new feature"
  3. 3

    Open a Pull Request

    Push your branch to GitHub and open a Pull Request (PR) to the `main` branch.

    git push origin new-feature
  4. 4

    Discuss and Review

    Team members review your code, leave comments, and suggest changes.

  5. 5

    Merge

    Once approved, merge the Pull Request into the `main` branch. Your changes are now part of the main codebase!
